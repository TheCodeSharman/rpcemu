# CLAUDE.md

This file guides Claude Code (claude.ai/code) when working in this
repository.  It lives on the `upstream` branch (see below) so it is
present on every branch and stays out of the patches we send upstream.

## What this is

A customised git fork of **RPCEmu**, the Acorn Risc PC / A7000 emulator.
Upstream RPCEmu is **Mercurial** at marutan.net and we can't push back, so
this fork keeps our patches as git branches and we send them to marutan as
unified diffs.  The RiscPc "raster-lab" project consumes this fork
(`setup-rpcemu.sh` clones it).

## Branch model

We don't control upstream, so the branch model is built around producing
**clean upstream-submission diffs** while still giving consumers one
predictable branch.

- **`upstream`** — RPCEmu upstream verbatim **plus git-conversion infra**.
  The verbatim import is tagged with the release version (`v0.9.5`,
  `v0.9.6`, ...); on top of that tag sit a few commits that exist *only*
  because we converted hg→git and are not meaningful upstream changes:
  `.gitignore`, untracking the runtime `cmos.ram`/`rpc.cfg` (the emulator
  regenerates both with defaults), `tools/reintegrate.sh`, and this
  `CLAUDE.md`.  Keeping them here keeps them **out** of the
  feature-vs-upstream diffs.  Changes only on a new import (or infra tweak).
- **`integration`** — the **derived integration branch** (the repo's
  **default** branch).  It is `upstream` + one **squashed** commit per
  feature branch, regenerated by `tools/reintegrate.sh` — never
  hand-edited.  This is what `setup-rpcemu.sh` clones and what consumers use.
- **`feature/*`** — **long-lived patch branches** off `upstream`.  Each
  carries one logical patch and has a **standing PR open against
  `upstream`** (the design-discussion thread / "what we'd send upstream"
  preview).  Never merged into `upstream`; `integration` picks them up via
  re-integration.

The submission diff for any patch is `git diff upstream feature/X` — it is
pure RPCEmu source, ready to email to marutan via the hg patch workflow.

### Re-integrating

`integration` is **rebuilt, not accumulated**.  Run `tools/reintegrate.sh`
(it lives on `upstream` so it survives the rebuild): it does
`git checkout -B integration upstream`, then `git merge --squash` + commit
for each feature in its `FEATURES` list, collapsing each branch to one
commit.  Review, then `git push --force-with-lease origin integration`.
Re-run any time a feature changes; add/remove a feature by editing the
`FEATURES` list.  `git rerere` is enabled, so a cross-feature conflict is
resolved once and replayed on later rebuilds.

### Importing a new upstream release

1. `git checkout upstream && git checkout -b sync/rpcemu-x.y.z`
2. Rsync new upstream source over the working tree (from a sidecar hg
   clone in `~/opt/rpcemu-upstream/`)
3. Commit as `Import RPCEmu x.y.z`, tag as `vx.y.z`, then re-apply the
   git-conversion infra commits (`.gitignore`, untrack, `reintegrate.sh`,
   this `CLAUDE.md`) on top
4. Open PR `sync/...` → `upstream`, merge once verified
5. For each `feature/*`: `git rebase upstream`, force-push; resolve any
   conflicts where upstream and our patches collide
6. Rebuild `integration` with `tools/reintegrate.sh`, then
   `git push --force-with-lease origin integration`

## Building

Qt5 / qmake build, from `src/qt5/`:

```bash
cd src/qt5
qmake -o Makefile rpcemu.pro    # only needed after editing rpcemu.pro
make -j"$(nproc)"
```

The default target is the **interpreter** (`rpcemu-interpreter`), written to
the **repo root**.  The recompiler target is `qmake CONFIG+=dynarec ...`
(→ `rpcemu-recompiler`).

**Use the interpreter.**  The dynamic recompiler is unstable on this
setup — RISC OS throws spurious errors under it (e.g. "no such SWI") because
the dynarec mistranslates the instruction stream; the interpreter runs
clean and is also the more timing-accurate of the two.

## Running

- Put a ROM in `roms/` — a single 4 MB `.rom` is a valid image on its own;
  RPCEmu concatenates everything in `roms/` alphabetically.
- `cmos.ram` (CMOS NVRAM) and `rpc.cfg` (GUI config) are **untracked and
  auto-generated** with sensible defaults if absent — they are local
  runtime state, not source.  `hd4.hdf` is a local hard-disc image.
- On a Wayland desktop, run the Qt5 binary with `QT_QPA_PLATFORM=xcb`
  (and ensure `libxcb-cursor0` is installed).
